# 网站最终规范化与维护工具重构计划

## Summary
- 采用 `using-superpowers` 与 `writing-plans` 流程，当前这轮先把网站展示规范彻底收口，再重构维护工具；网站端不再继续“头痛医头”，而是补一层明确的“规范显示模型”，让前端展示和底层脏数据解耦。
- 已确认当前网站问题的根因主要有两类：
  - 原始数据存在事实错误、字段含义混用、全名/俗名/英文名/原文名混填、乐团与人物简称缺失、部分条目年份与国家错误。
  - 前端展示层和搜索索引层缺少统一的规范显示规则，导致搜索页、目录页、作品页对同一实体展示不一致。
- 已确认当前自动检查不可用不是单点 bug，而是架构问题：现在是同步直连外网、无任务状态、无进度、无诊断、无失败分层，所以用户只能看到 `fetch failed`，无法判断是否在运行、卡在哪、影响了哪些条目。

## Key Changes
### 1. 网站端最终规范化
- 为 `Composer` 与 `Person` 增加一组专用规范显示字段，作为网站与搜索的唯一展示来源：
  - `displayName`: 规范中文俗名，作为一级显示名。
  - `displayFullName`: 规范中文全称。
  - `displayLatinName`: 规范英文/拉丁/母语全称。
  - `abbreviations`: 简称与缩写，尤其用于乐团和组合，例如 `VPO`。
- 保留现有 `name/fullName/nameLatin/aliases` 作为原始或兼容字段，但前端不再直接依赖它们做最终展示。
- 搜索索引统一改为“规范双层名”策略：
  - 搜索结果继续保持当前的分组列表样式，不升级成重卡片。
  - 每条结果固定展示 `displayName + displayFullName / displayLatinName`，不再出现有的只显示姓、有的显示全名。
  - 乐团搜索支持中文、英文、原文、简称、缩写、常见别称；搜索 token 中纳入 `abbreviations`、规范字段、别名、旧名称。
- 目录页与搜索页统一使用同一套显示规则，不再出现“目录页较完整、搜索页过简”的分裂。
- 作品页推荐版本条目改为规范化渲染，不再直接显示 `recording.title`：
  - 一级标题按固定顺序拼接：`指挥俗名 - 乐团简称 - 艺术家俗名 - 组合简称`。
  - 二级信息展示英文/原文全称与乐团全称。
  - 时间与地点保留在副标题中。
  - 删除所有条目里重复的“查看完整演出信息、外部资源链接与图片。”文案。
  - 若缺少指挥，则按同一规则回退到主艺术家/组合/乐团，确保每条仍有稳定标题。
- 把当前网站暴露出的数据问题显式记录成“维护工具待修复问题类型”，至少覆盖：
  - 错误国家/年代
  - 中英文名错配
  - 俗名与全名混用
  - 人物与乐团字段串位
  - 作品/代表作错误
  - 版本条目标题不一致
  - 简称、缩写、中文别名缺失

### 2. 搜索与索引层重做
- 重构 `searchIndex` 的数据结构，不再只存 `title/subtitle/keywords`，而是明确区分：
  - 主显示文本
  - 副显示文本
  - 规范匹配词
  - 别名/缩写词
  - 类型标签
- 增加“模糊但可解释”的匹配规则：
  - 规范化大小写、空格、连字符、点号、全半角。
  - 支持中英混搜、缩写命中、简称命中。
  - 乐团和人物允许通过简称与缩写命中，如 `VPO`、`BPO`。
- 搜索页仍保持分组式结果，但分组内展示统一的规范文本，且为乐团补上中文可搜与中文展示。

### 3. 维护工具重构为真正的任务流
- 自动检查从“同步请求”改为“异步 job 模型”：
  - 创建检查任务
  - 后台逐条执行
  - 前端轮询或流式读取状态
  - 完成后生成候选 run
- 自动检查页改成完整 dashboard，而不是单个按钮加结果框：
  - 运行状态：未开始、准备中、运行中、部分失败、已完成、已取消。
  - 总进度条：按条目数推进。
  - 当前处理项：展示当前条目、当前阶段、当前来源。
  - 统计概览：总条目、成功、失败、待人工确认、跳过。
  - 错误诊断：来源站点、HTTP 状态、超时、解析失败、无命中、LLM 不可用等分开显示。
- 自动检查选取逻辑改为“分面筛选 + 选取预览 + 手动反选”：
  - 保留按类别筛选：作曲家、指挥、乐团、艺术家、版本。
  - 范围限定从单个下拉框升级为多维过滤：按作曲家、作品、人物角色、乐团、版本交叉筛选。
  - 在筛选区下方增加“受影响条目预览栏”，按类别显示紧凑 chips，支持点选取消与重新加入。
  - 自动检查前必须先显示最终选中集合，避免黑箱运行。
- 在单条详情页顶部增加“自动检查此条目”按钮：
  - 支持单独为当前作曲家/人物/作品/版本发起一次检查。
  - 检查完成后直接进入该条目的候选 diff 审查。
- 候选审查页改为“状态驱动”的确认工作台：
  - 条目状态至少包含：未查看、已查看、已修改未确认、已确认、已放弃。
  - 支持字段级 diff、图片前后对比、单条确认、单条放弃、批量确认、批量放弃。
  - 最终有统一“应用全部已确认修改”和“放弃本次检查”的入口。
- 日志与回滚页保留并增强：
  - 每次 run 保留完整日志、范围、来源、修改字段、图片变化、失败原因、合并建议。
  - 支持单条回滚和整次 run 回滚。
- 自动检查失败时不得只回 `{ error: "fetch failed" }`：
  - 网络错误、来源错误、解析错误、无结果、权限错误、速率限制都要结构化返回。
  - 即使部分来源失败，也要保留已完成结果和已处理进度，不能整次直接白屏。

### 4. 自动检查与 LLM 辅助策略
- 保持“候选先行，不直写正式数据”的安全策略不变。
- 规则引擎仍是主干，LLM 只做辅助，不做最终裁决。
- LLM 接入固定采用 `OpenAI-compatible` 配置方式：
  - 可配置 `base URL`、`API key`、`model`。
  - 提供独立“连接测试”与“模型测试”按钮。
  - 若未配置或不可用，自动回退到纯规则工作流，不阻塞工具使用。
- LLM 仅用于高模糊任务：
  - 对人物/乐团/版本候选结果做二次比对说明。
  - 为缺失简介的条目生成简短简介候选。
  - 为疑似重复人物或乐团给出“为何疑似同一实体”的解释。
- LLM 绝不直接写正式字段：
  - 只生成候选说明、候选摘要、候选匹配理由。
  - 最终仍由候选审查页确认后应用。
- 自动检查流程固定为：
  1. 解析筛选条件并生成最终条目集合
  2. 对外部来源做可用性探测
  3. 逐条抓取结构化来源
  4. 规则引擎生成字段候选、图片候选、合并候选
  5. 若已配置 LLM，则对高风险或高模糊项做辅助判断
  6. 将结果写入候选 run
  7. 在审查页确认后再应用
- 图片自动化沿用上一轮思路，但纳入新任务流：
  - 人物/乐团优先 `Wikipedia / Wikidata / Wikimedia Commons`
  - 版本优先结构化专辑封面与官方发行图
  - 图片只进入候选，不直接落正式字段
  - 应用时才本地保存并记录来源元数据

### 5. 接口与数据结构调整
- 网站数据模型新增规范显示字段：
  - `displayName`
  - `displayFullName`
  - `displayLatinName`
  - `abbreviations`
- 自动检查接口从“单次同步返回 run”升级为任务型接口：
  - 创建 job
  - 查询 job 状态
  - 查询 job 进度与阶段日志
  - 取消 job
  - 基于单条实体创建 job
  - 查询筛选预览结果
  - 测试 LLM 配置
- run/proposal 状态模型增加面向审查的状态字段，而不只保留 `pending/applied/ignored`。
- 本地配置新增未跟踪的 owner-tool 设置文件，用于保存 LLM provider、base URL、model、API key 与超时策略；未配置时功能可降级。

## Test Plan
- 网站展示：
  - 搜索结果中作曲家、人物、乐团一律显示规范双层名，不再出现只显示姓或只显示英文的条目。
  - 乐团支持中文搜索、英文搜索、简称搜索、缩写搜索。
  - 作品页所有推荐版本条目标题规则一致，缺失字段时按既定回退策略生成，不出现一条带年份、一条不带年份的混乱标题。
  - 版本条目中不再出现重复 boilerplate 文案。
- 数据规范层：
  - 为人物与乐团新增的规范显示字段有校验测试。
  - 搜索索引测试覆盖中文、英文、原文、简称、缩写命中。
  - 目录页与搜索页对同一实体展示文本保持一致。
- 自动检查任务流：
  - 创建 job 后能看到准备中、运行中、完成或失败状态。
  - 部分来源失败时，任务仍能保留已完成进度与局部结果。
  - 筛选预览与最终实际处理条目集合一致。
  - 单条实体自动检查和批量自动检查都能产出候选 run。
- 审查与回滚：
  - 候选状态流转覆盖未查看、已查看、已修改未确认、已确认、已放弃。
  - 单条确认、批量确认、单条回滚、整次回滚都要有端到端验证。
  - 图片候选应用后能正确落本地文件并记录来源，回滚后引用恢复。
- LLM 接入：
  - 无 API key 时界面明确提示“已降级为纯规则模式”。
  - 配置可用 OpenAI-compatible 服务时，连接测试与模型测试都返回结构化结果。
  - LLM 失败、超时、返回空结果时不会阻塞整次自动检查。
- 用户可用性回归：
  - 不能只在测试里通过，必须用“实际启动 owner tool 后点选自动检查”的方式验证一次完整交互。
  - 特别验证你这次指出的场景：用户点击自动检查后，界面必须能看到选取范围、进度、当前条目和错误原因，而不是只剩一个失败 JSON。

## Assumptions
- 命名规范采用你已确认的“规范双层名”策略：一级显示规范中文俗名，二级显示中文全称与英文/原文全称。
- 搜索页保留当前的分组式结果，不改成重卡片。
- LLM 首版采用 OpenAI-compatible 接口，而不是只绑定官方 OpenAI。
- 规范显示字段由维护工具负责维护与校正，网站前端只消费规范字段，不再直接拼接原始字段。
- 自动检查、LLM 建议、图片抓取、合并建议都只能生成候选，正式数据必须经过审查确认后再应用。
